# Badcase 看板使用说明

## 为什么要做 Badcase 看板

AI 产品的质量不是靠一次性“生成得像样”来保证，而是靠持续记录失败案例、定位根因、修复并复测来提升。

本项目的 Badcase 看板用于记录：

- 判断不准
- 建议过泛
- 漏判
- 隐私风险
- 模型幻觉风险

## 一条 Badcase 应该包含什么

| 字段 | 含义 |
| --- | --- |
| ID | 例如 BC-001 |
| 日期 | 发现问题的日期 |
| 样本 | 出问题的 JD/简历样本 |
| 类型 | 误判、建议过泛、漏判、隐私风险、幻觉风险 |
| 状态 | 待修复、观察中、已修复 |
| 现象 | 用户看到的具体问题 |
| 根因 | 为什么会发生 |
| 修复方案 | 产品或技术上怎么改 |
| 验证方式 | 如何确认它真的变好了 |

## 后续迭代方式

每次你测试一个样本，都按下面四个问题记录：

1. 判断准不准？
2. 建议是否具体、可执行？
3. 有没有漏掉关键能力？
4. 有没有隐私、幻觉或表达误导风险？

如果发现问题，就新增一条 Badcase，而不是只在聊天里说“这里不太对”。这会让项目越来越像真实 AI 产品。

## 当前产品内反馈闭环

页面中的 Badcase 看板已经支持本地新增反馈：

1. 进入“Badcase”页签；
2. 选择问题类型；
3. 填写现象、根因、修复方案和验证方式；
4. 点击“新增 Badcase”；
5. 记录会保存到浏览器 localStorage。

## 为什么先用 localStorage

当前阶段优先保证工具开箱可用，不依赖后端、不需要登录，也不上传用户简历或反馈内容。

后续开源协作可以升级为：

- 导出 Badcase JSON；
- 生成 GitHub Issue 模板；
- 接入后端数据库；
- 支持团队协作标注。

## 导出与贡献

当前页面支持两种导出：

1. 导出 JSON  
   用于备份、批量分析或后续导入数据库。

2. 生成 Issue  
   用于复制到 GitHub Issue 中，方便开源协作者一起讨论和修复。

建议优先提交“待修复”和“观察中”的 Badcase。已经修复的问题可以保留在项目中作为迭代记录。
